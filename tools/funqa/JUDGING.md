# Fun QA 판정 프로토콜 (Phase 2)

Phase 1 정량 메트릭이 못 잡는 "재미"를 Claude Code 개발 세션이 정성 판정한다. 외부 API·비용 0.

## 절차
1. `node tools/funqa/judge-pack.cjs [N] > funqa-judge-pack.md` — 판정 입력 생성.
2. Claude Code 세션에 "이 판정 팩 읽고 재미 판정해줘" → 팩을 읽고 판정.
3. 결과를 `tools/funqa/judgment-YYYY-MM-DD.md`로 저장 (Phase 3 버전 비교 기준선).

## 판정 기준
- **취향 렌즈로 본다**: 각 페르소나가 *무엇에 재미를 느끼고 무엇에 이탈하는지*(팩의 판정 렌즈)를 기준으로.
- **⚠️ 정량 점수에 휘둘리지 말 것**: 5축이 비슷하거나 높아도, 궤적 내러티브가 실제로 같은/재밌는 경험인지 독립 판단.
- 자주 나오는 함정:
  - **낙승(마진 큼) = 긴장 없음**: 도파민(큰 폭발) 점수가 높아도, 매 라운드 마진이 1.5+ 면 "자동조종"이라 공허. "빠듯한 마진"(핵심 재미)이 죽는다.
  - **운빨 죽음 = 주체성 없음**: 낙승하다 보스에서 손패 안 풀려 갑자기 죽으면, 수읽기로 막을 수 없는 죽음 = 좌절.
  - **페르소나 궤적이 동일 = 차별 실패**: 두 페르소나의 대표 궤적이 글자 그대로 같으면, 5축 수렴이 "실질적으로 같은 플레이"라는 증거.
  - **클리어 0% + 정량 높음**: 영원히 못 깨는 건 결국 좌절. 흐름 축이 이를 충분히 반영 못할 수 있음.

## 출력 형식
- 페르소나별: **재미 0~10 + 한 줄 이유 + 의심지점**(Phase 3에서 think-aloud로 재생).
- 종합: **대중재미 정성판정**(패널 70%+ 통과?) + Phase 1 정량과 **일치/괴리** 명시.

## 보완 제안 도출 (개발 피드백 루프)
판정은 "측정"에서 끝나지 않는다. 발견한 재미 문제마다 **게임 보완 액션**으로 연결한다 — 보고서가 곧 개발 세션의 보완 입력이 되도록.
- 예: "낙승 일색(마진 큼) → `blindTarget` 상향 또는 마진 압축 검토"
- 예: "운빨 즉사(보스에서 손패 안 풀림) → 안테1 보스 풀 완화 / 손패 회복 수단"
- 예: "콤보러·캐주얼 클리어 0% → 후반 목표(`blindBase` 지수) 완화 또는 비최적 플레이 허용폭"
- ⚠️ 게임 코드(`index.html`) 직접 수정은 **사용자 승인 후**. 보고서는 제안까지.

## 회귀 비교 (보완 효과 측정 — 루프의 닫힘)
- 보완 전: `node tools/funqa/regress.cjs --save` → 현재 게임 상태를 기준선으로 저장.
- 보완 후: `node tools/funqa/regress.cjs` → 페르소나별 재미 Δ + 클리어% Δ. 재미 하락(Δ≤-0.3) 시 exit 1(게이트).
- "이 보완이 재미를 정말 올렸나"를 숫자로 확인 → 다음 보완 반복.
