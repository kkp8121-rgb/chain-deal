# 부적 해금 시스템 (Charm Unlock System) — 설계

> 최종 업데이트: 2026-06-19 · 대상 버전: v3.15 예정
> PLAN.md C3(해금/컬렉션)의 첫 단계 = **부적 해금**. HANDOVER §7 🥈 "보상/리텐션 메타(다시 하는 이유)".

## 0. 한 줄 요약

B1(v3.14)에서 추가한 **5개 신규 아키타입 부적을 "벌어서 여는" 콘텐츠로 전환**한다. 코어 경험은 첫 판부터 온전하고, 해금이 곧 "새 빌드 경로 개방"과 일치한다. 해금은 메타층(localStorage)이라 라운드 점수 규칙·밸런스 기준선은 불변.

## 1. 콘텐츠 — 시작 세트 vs 잠금

| 구분 | 부적 (id) | 근거 |
|---|---|---|
| **시작 해금 (5)** | 탐욕(greed)·발화(pyro)·플러시광(suited)·러너(runner)·잭팟(jackpot) | v3.14 이전 코어 5종. 새 플레이어가 첫 판부터 빌드 가능 |
| **잠금 (5)** | 흑심(noir)·중개상(broker)·쌍둥이(twins)·정련가(compactor)·잔챙이(runts) | 정확히 B1 신규 아키타입. 잠긴 게 전부 "확장"이라 코어 밸런스 기준선(그리디 클리어율) 불변 |

## 2. 해금 트리거 (혼합) — 전부 `settle()` 시점에 판정 가능

> ★설계 제약: 모든 조건을 **정산(settle) 순간에 `cd_stats`/`S.row`/`evalHand()` 로 즉시 판정**. 런 중 이벤트 추적용 새 상태 변수 0개 — 단순성 보존.

**등급 뼈대 (cd_stats 재사용 · 기존 플레이어 자동 소급)**
- **흑심(noir)** ← 안테 4 도달 (`bestAnte >= 4`)
- **정련가(compactor)** ← 첫 런 클리어 (`wins >= 1`)

**도전과제 (부적 정체성을 가르치는 발칙 조건)** — `S.row` 의 **rank 개수**로 직접 판정(족보 이름의 텍사스 서열 비교가 아님 → 모호성 제거)
- **중개상(broker)** ← 한 라운드 줄에 **같은 숫자 3장 이상**(트리플+, `maxRankCount>=3`). broker가 보상하는 같은-숫자 족보군. (자연 발생, intro 해금)
- **쌍둥이(twins)** ← 한 라운드 줄에 **같은 숫자 2장 그룹이 2개 이상**(투페어+, `pairs>=2`). twins가 보상하는 "겹치는 그룹." (자연 발생, intro 해금)
- **잔챙이(runts)** ← 한 줄(8장)에 **A·2·3 저랭크 4장 이상**. 저랭크는 점수 손해라 그리디론 안 나옴 → **의도적 저랭크 플레이를 강요하는 진짜 도전**.

## 3. 데이터 / 영속성

- `localStorage["cd_unlocked"]` = 해금된 부적 id 배열. 기본값 = 시작 5종.
- **해금 메타 정의**: 각 잠금 부적에 `{cond: (stats,row)=>bool, hint: "해금 조건 문구"}` 를 매핑하는 `UNLOCKS` 테이블.
- **소급(grandfather)**: 로드/해금판정 시 `cd_stats`(bestAnte/wins) 기준 등급 조건을 재평가 → 이미 만족하면 자동 해금. 기존에 안테5 찍은 플레이어는 흑심을 즉시 보유.
- **게이팅**: `shopPool()` 에서 `CHARMS.filter(c => !has(c.id) && isUnlocked(c.id))` — 잠긴 부적은 상점에 안 뜸.

## 4. 해금 판정 시점

- `settle()` 의 stats 집계 직후 `checkUnlocks(stats, S.row)` 호출:
  1. 갱신된 `cd_stats` + 이번 라운드 `S.row` 로 각 잠금 부적 `cond` 평가
     - ★`stx.bestAnte` 는 런 종료시에만 갱신되므로, noir 즉시 해금을 위해 `checkUnlocks({...stx, bestAnte:max(stx.bestAnte, S.ante)}, S.row)` 로 **현재 도달 안테를 반영**(저장 stats·newRec 연출엔 영향 없음).
  2. 새로 충족된 id 를 `cd_unlocked` 에 추가·저장
  3. 해금된 부적 목록을 반환 → 정산 표/배너에 해금 연출

## 5. UI (기존 드로어 패턴 재사용)

- **컬렉션 드로어** (`openDrawer('charms')` 신규 type): stats 줄 또는 버튼에서 진입. 10종 그리드 —
  - 해금: 이름·효과 정상 표시
  - 잠금: 실루엣(효과 가림, 흐림 처리) + **해금 조건 힌트** 한 줄
- **해금 알림**: 정산 표(`#tally`)에 `🔓 새 부적 해금: 흑심!` 줄 추가 + 상승음(`beep`). 기존 `flash`/tally 인프라 재사용. 여러 개 동시 해금 시 모두 표기.

## 6. 영향 / 리스크

- 신규 플레이어 초반 상점이 5종 풀 → 희석 더 적음 = 초반 *살짝 쉬움*(무해, run-sim 기준 부적 10종 희석 −0.9%p 보다 작은 효과).
- **sim 도구 영향 없음**: 해금은 메타층이라 라운드 점수 규칙 불변. `run-sim.cjs` 의 "상점 희석" 수치는 전 부적 보유를 가정하므로 신규 플레이어엔 과대평가됨 — 코드 주석으로만 명시(시뮬 수정 불필요).
- 잠금이 코어 5종이 아니므로 `balance-check.cjs` 그리디 클리어율 기준선 불변.

## 7. 비범위 (YAGNI)

- 보스/카드효과/시작 덱 해금 — 차기. 이번엔 **부적만**.
- 메타화폐/아웃게임 경제(C2) — 별개 작업. 해금은 **순수 업적 기반**(갈아넣는 화폐 금지, HANDOVER §7 원칙).
- 해금 진행도 알림 배지 등 부가 UX — 후순위.

## 8. 검증 기준 (Definition of Done)

1. `node tools/balance-check.cjs` 문법 OK + 그리디 클리어율 기존 수치 유지(잠금이 메타층이라 불변이어야 함).
2. 새 플레이어(빈 localStorage): 시작 5종만 상점 등장 / 컬렉션 드로어에 잠금 5종 실루엣+힌트.
3. 각 해금 조건 충족 시 해당 부적이 cd_unlocked 에 추가되고 이후 상점에 등장.
4. 기존 플레이어(bestAnte≥4, wins≥1 저장됨): 로드 시 흑심·정련가 자동 소급 해금.
5. 해금 순간 정산 표에 알림 + 상승음.
