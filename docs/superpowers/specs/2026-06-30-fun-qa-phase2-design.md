# Fun QA — Phase 2 (LLM 재미 판정관) 설계

작성일: 2026-06-30
상태: 설계 → plan
선행: Phase 1 (정량 코어) — main 머지 완료, 전체 테스트 통과

> **정의** — Phase 1이 산출한 정량 메트릭과 대표 플레이 궤적을 입력으로, **Claude Code 개발 세션이 페르소나별 "재미"를 정성 판정**하는 Layer 3. 외부 API·비용 0(개발 세션 자체가 판정자), 외부의존성 0 유지.

---

## 1. 확정 결정

- **LLM 호출 = Claude Code 세션 판정** (API 자동호출 아님). 이유: 이 프로젝트는 "외부 라이브러리 0·빌드 0" 원칙 + public 리포(시크릿 위험) + 이미 Claude Code로 개발 중. API 키·네트워크·비용 없이 개발 세션이 곧 판정자.
- **입력 형태 = 마크다운 판정 팩** (수동 판정). slash 커맨드 자동화는 YAGNI(나중에).

## 2. 배경 — Phase 1이 남긴 것

Phase 1 첫 리포트: **대중 재미 60% → FAIL**. 그리고 정량 메트릭이 못 잡은 두 가지:
1. **마스터리·안전·스릴 5축 수렴** — 1-ply 배치라 세 페르소나가 거의 같은 플레이 → 메트릭으로는 구별 안 됨.
2. **콤보러·캐주얼 클리어 0%인데 재미점수 높음** — 흐름 축이 클리어 실패를 충분히 반영 못함.

→ Phase 2의 핵심 가치: **숫자로 수렴해 보이는 것이 "정말 같은 재미인가"를 LLM 정성판단으로 가른다.**

## 3. 구성요소

1. **`tools/funqa/judge-pack.cjs`** — 판정 입력 생성기. `node tools/funqa/judge-pack.cjs [N] > funqa-judge-pack.md`
2. **`tools/funqa/JUDGING.md`** — 판정 프로토콜/지침 (Claude Code가 어떻게 판정하는지)
3. **시범 판정** — Claude Code가 실제 팩을 읽고 5종 판정 1회 = Phase 2 검증

## 4. 판정 팩 구조 (`funqa-judge-pack.md`)

```
# Fun QA 판정 팩
## Phase 1 정량 요약
  5종 재미점수 표 + 대중재미 %  (judge가 정성판단과 대조할 기준선)

## 페르소나: 마스터리
  - 재미 5축: agency 0.77 / tension 0.21 / ...
  - 판정 지침: "이 페르소나는 [취향]을 중시하고 [X]에 이탈한다. 그 렌즈로 봐라."
  - 대표 궤적 3종:
    ### 전형적 승리 (마진 중앙값 판)
      안테별 내러티브
    ### 전형적 패배 (최빈 죽음 안테)
      안테별 내러티브 + 죽음 지점
    ### 아슬아슬 near-miss
      안테별 내러티브
  (× 5 페르소나)

## 판정 요청 (Claude Code가 채울 빈칸)
  각 페르소나: 재미 0~10 + 이유 + 의심지점
  종합: 대중재미 정성판정 + Phase1 정량과 일치/괴리
```

## 5. 샘플 궤적 선별 기준

각 페르소나의 N판에서:
- **전형적 승리**: 클리어한 판 중 최종 마진의 **중앙값**에 가까운 판.
- **전형적 패배**: **최빈 죽음 안테**에서 죽은 판 (가장 흔한 실패 양상).
- **near-miss**: 마진 1.0~1.15 통과 라운드가 가장 많은 판 (아슬아슬한 손맛).

## 6. 궤적 내러티브 포맷 (LLM 가독)

라운드별 1줄 요약(턴 전체 X — 너무 큼):
```
안테1 작은: 클리어 (마진 1.40, 최대체인 3, 하이카드)
안테1 큰:   클리어 (마진 1.21, 최대체인 4, 페어)
안테2 보스: 클리어 (마진 1.08, 최대체인 5, 플러시)  ← 아슬아슬
안테3 보스(인색한손): 실패 (점수 280/목표 340, 손패2)
```
한 라운드 = {안테/블라인드, 통과여부, 마진, 최대체인(maxRunLen), 족보(handKind), 보스, (패배 시)점수/목표}.

## 7. 판정 프로토콜 (`JUDGING.md`)

1. `node tools/funqa/judge-pack.cjs > funqa-judge-pack.md`
2. Claude Code에 "이 판정 팩 읽고 재미 판정해줘" → 페르소나별 0~10 + 이유 + 의심지점.
3. 종합: 대중재미 정성판정 + Phase 1 정량(60% FAIL)과 **일치하는가/괴리하는가** 명시.
4. 판정 지침: 정량 점수에 **휘둘리지 말 것** — 5축이 같아도 궤적 내러티브가 실제로 같은 경험인지 독립 판단.

## 8. 검증 (Phase 2 완료 기준)

- `judge-pack.cjs`가 5종 페르소나 × 3궤적 마크다운을 정상 생성 (NaN·빈 궤적 없음).
- Claude Code 시범 판정 1회 성공 — 페르소나별 점수+이유+의심지점 산출.
- **핵심**: LLM이 "마스터리·안전·스릴 5축 수렴"을 *"실질적으로 같은 플레이 → 페르소나 차별 부족"* 으로 짚어내는가 (정량이 못 잡은 통찰).

## 9. 비목표 (Phase 3)

- 판정 결과 저장 + 버전 비교 회귀 게이트.
- LLM 플레이테스터(think-aloud, 의심지점 직접 재생).
- 스테이크 가변.
