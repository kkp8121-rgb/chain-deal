# Fun QA — AI 자동 재미 검증 시스템 (설계)

작성일: 2026-06-30
상태: 설계 승인 대기 → writing-plans
대상: CHAIN DEAL (현재 구현 `prototype/index.html` v3.x)

> **한 줄 정의** — 패치마다, 다양한 취향의 **AI 페르소나 봇**들이 CHAIN DEAL을 수천 판 자동 플레이하고, 그 궤적에서 **재미 신호를 정량 추출**한 뒤 **LLM이 사람 디자이너처럼 해석**하여, "**대다수 유저가 재밌어 하는가**"를 **분포로 판정**하고 구체적 진단을 내는 개발 게이트.

---

## 1. 배경 — 왜 만드는가

### 1.1 메우려는 구멍: "시뮬은 멀쩡한데 사람 체감은 가혹"
기존 시뮬(`run-sim.cjs`, `strategy-sim.cjs`)은 봇 그리디 자동플레이로 **클리어율·승률·점수·사망지점** 같은 *난이도* 지표만 측정한다. 그러나 핵심 교훈(안테1 인색한손 보스 사례)은 — *"손패 감소가 점수가 아니라 **선택지(주체성)**를 깎았고, 봇 그리디 클리어율(61%)은 멀쩡한데 사람 체감은 가혹했다 → 시뮬이 못 잡는 체감"* 이었다. **즉 난이도는 잡았지만 재미는 못 잡았다.**

### 1.2 이미 있는 철학: GDD §9.2 재미 게이트
GDD는 이미 *"헤드리스 자가검증 — 도파민 지표 1급화"*를 명문화했다:
- 죽은 탭 비율 < 15%
- 라운드당 연쇄≥3 이벤트 ≥ 2회 (도파민 코어)
- 무작위 connect율 30~40%
- 캐주얼-진지 점수 격차 < 50%
- 발산 시드 0%

**Fun QA는 이 철학을 실제로 구현 + 페르소나 패널 + LLM 정성판정으로 확장하는 것이다.** (단 GDD의 일부 게이트는 미구현 메커닉 기준이므로, 현재 구현된 게임에 맞게 재정의한다 — §4 참조.)

### 1.3 재미는 주관적이다 → 분포로 측정
재미는 사람마다 다르다. 단일 봇·단일 평가자로는 무의미하다. 발라트로는 *넓은 취향대*를 만족시켜 성공했고, 폐기한 STRANDLOOM은 소수 퍼즐 취향만 겨냥해 실패했다. **목표는 "재미 점수 하나"가 아니라 "재미의 분포"이며, 판정 기준은 "대다수(페르소나 패널 70%+)가 재밌어 하는가"다.**

## 2. 목표 / 비목표

### 목표
- 게임 코드(`index.html`)를 **손대지 않고 옆에 격리(greybox)된** `tools/funqa/` 도구로 구축.
- 패치마다 돌릴 수 있을 만큼 **싸고 빠른** 정량 코어(LLM 없이) + 필요 시 **LLM 정성판정** 2층 구조.
- 출력은 **대중 재미 점수 + 페르소나별 강약 + 구체적 진단 + 패치 회귀 + pass/fail 게이트**.

### 비목표
- 사람 플레이테스트 대체 (보조·선별 도구이지 최종 판정자 아님).
- 게임 규칙 자체의 리팩터/개선 (선결 인프라 §7 외 게임 로직 변경 금지).
- 실시간 인게임 AI (개발 루프 전용, 런타임 X).

## 3. 핵심 통찰 — 왜 봇으로 재미가 잡히나

- 게임의 **유일한 인-라운드 결정 = "손패 3장 중 1장 선택"** (`placeCard(hi)`, `hi ∈ [0, handSize)`). 카드는 줄 끝(왼→오, 8칸)에만 붙고 좌우 선택이 없다. 이 **단일 결정점이 페르소나가 갈리는 정확한 지점**이다.
- "주체성"은 측정 가능하다: 매 턴 손패 후보들의 **즉시 점수 분포**에서 `best - second` 격차가 0에 가까우면 = 뭘 골라도 똑같음 = 지루(= 안테1 보스가 깎은 것). 봇으로도 이걸 잡는다.
- 기존 `strategy-sim.cjs`의 pluggable `pick(hand, row, boss)` 콜백이 페르소나 봇의 정확한 확장점이다.

## 4. 아키텍처 — 3층 (비용 점증)

```
Layer 3  LLM 재미 판정관 (질, 비용 발생) ── 페르소나별 정성평가 + think-aloud 검증
   ↑ 메트릭 + 샘플 궤적
Layer 2  재미 메트릭 (정량 프록시, 무료) ── 봇 궤적 → 재미 5축
   ↑ 플레이 궤적(로그)
Layer 1  페르소나 봇 (양, 거의 무료) ── 5종 × 수천 판 × 시드 고정
```

### Layer 1 — 페르소나 봇
5종 페르소나 = `배치정책(pick) × 상점정책(STRATS) × 리스크/노이즈 파라미터` 조합:

| 페르소나 | 배치 정책(pick) | 상점 정책(STRATS) | 리스크/노이즈 | 무엇에 재미·이탈 |
|---|---|---|---|---|
| **콤보러** | `pickRank`/`pickFlush` 파생 | cartel·gem | 고분산 추구 | 큰 폭발↑ / 밋밋한 판↓ |
| **안전제일** | 마진우선(ctx 기대값) | balance | 저분산 | 안정 진행↑ / 운빨 억까↓ |
| **스릴러** | 룩어헤드 잭팟 | apex·jackpot | 클러치 도박 | 아슬아슬↑ / 너무 쉬움↓ |
| **캐주얼** | Top-K 랜덤(노이즈) | 즉흥 | 무관심 | 직관적 손맛↑ / 수읽기 강요↓ |
| **마스터리** | 풀 룩어헤드 best | 최적 | 정밀 | 깊은 수읽기↑ / 얕은 선택↓ |

- 규칙 엔진은 **최신 `run-sim.cjs`에서만** 복사한다(구버전 `balance-check`/`strategy-sim`/`hand-frequency`는 부적 13종 이전·fiveKind 미반영이므로 엔진 소스 금지 — 메트릭 집계 패턴만 참조).
- 시드 고정(`mulberry32`)으로 **동일 국면에서 페르소나 A/B 공정 비교** 가능.

### Layer 2 — 재미 메트릭 (5축)
봇 궤적에서 추출. 각 축은 원시 지표 + 0~1 정규화 점수. (각 페르소나는 5축에 다른 가중치를 둔다.)

1. **주체성 (Agency)** — `placeCard` 후보들의 즉시 점수 분포에서 `(best−second)/best`. *적당히 클 때* 높음(선택이 의미). 0이면 지루(억까 보스), 항상 동일 큰 값이면 "정답 뻔함"(가짜 주체성). 합법수 의미성 = 후보 결과 분산.
2. **긴장 (Tension)** — near-miss율(클리어 마진 `score/target` ∈ [1.0, 1.15] 통과 비율). "빠듯한 마진"(핵심 재미)의 정량화. blowout(≥1.5 너무 쉬움)·즉사(<0.5 손쓸수없음)는 감점.
3. **도파민 (Dopamine)** — 짜릿순간 밀도(라운드당 `runLen≥4` OR 희소족보(flush 이상) 횟수, GDD 목표 ≥2). 폭발 타이밍 분포(후반 7·8칸에만 몰리면 "도화선" 문제 = 감점).
4. **다양성 (Variety)** — 승리 빌드(부적 조합)의 엔트로피, 지배전략 존재 여부(한 빌드만 항상 이기면 노잼). 페르소나 간 결과 다양성(같은 시드를 다르게 푸는가).
5. **흐름 (Flow)** — 안테별 사망률 곡선(완만 단조 증가 vs 특정 안테 절벽). 좌절(주체성 낮은 채 사망) 비율, 압도(긴장 0 구간) 비율. GDD "캐주얼-진지 격차 < 50%" 흡수.

### Layer 3 — LLM 재미 판정관
- **페르소나 판정관**: 각 페르소나의 5축 메트릭 + 샘플 궤적 3~5개(안테별 선택·폭발·죽음 요약)를 받아, *그 취향 유저 시각*에서 재미점수(0~10) + "왜" + 구체적 문제 + **의심지점**(검증 필요 시드)을 반환.
- **LLM 플레이테스터 (think-aloud)**: 의심지점 시드를 LLM이 직접 매 턴 플레이하며 "지금 재밌다/지루하다/억까당했다"를 자기보고. 비용이 크므로 **소수 정예**(주요 패치·의심지점만).
- 영상 원칙 적용: 정확일치 아닌 **의미적 평가**, 허용 범위, 비용·속도 트레이드오프 명시.

---

## 5. 출력 — Fun QA 리포트

- **대중 재미 점수**: 페르소나 5종 재미점수 분포. **70%+ (5종 중 4종+)가 임계(예: 6/10) 이상이면 PASS.**
- **편향 경고**: 특정 페르소나만 높고 나머지 낮으면 "소수취향" 플래그 (예: "콤보러 9 / 안전러 3 → 대중성 위험").
- **구체적 진단**: "안테1이 전 페르소나에서 지루 — 주체성 0.1" 같은 행동 메트릭 근거 + LLM 해석.
- **패치 회귀**: 직전 버전 대비 페르소나별 재미 델타 ("이 변경이 안전러 재미 −30%").
- **pass/fail 게이트**: `economy-check.cjs`의 `assert` + `process.exit` 패턴으로 CI/수동 게이트화.

## 6. 핵심 검증 (골든 테스트) — "메트릭이 알려진 사례를 잡나"
설계의 사활: **재미 메트릭이 이미 아는 재미 붕괴를 잡아내야 한다.**
- **안테1 인색한손 보스(손패3→2)** 시드에서 → 주체성 메트릭이 유의하게 낮게 나와야 한다(메모리의 체감 사례를 정량 재현). 잡으면 메트릭 유효, 못 잡으면 메트릭 재설계.
- 가능하면 텔레메트리(`logEvent`/구글시트) 사람 플레이 데이터(중도이탈·리트라이 지점)와 봇 메트릭을 대조해 캘리브레이션.

## 7. 선결 인프라 (sim-infra가 짚은 부채)
Fun QA를 풀런에 태우려면 `run-sim.cjs`에 최소 변경 3가지가 필요하다(게임 `index.html`은 불변):

- **(A) pick 파라미터화** — `playRound`(L98)의 인라인 그리디 루프(L104-106)를 `pick` 콜백 인자로 교체 → `strategy-sim`의 `runRound` 패턴과 통일. 페르소나를 풀런에 태우는 단일 최소 변경.
- **(B) ctx 확장** — `pick(hand, row, boss, ctx)`로 시그니처 확장, `ctx = {score, target, slotsLeft, deckSize}`. 안전/스릴 페르소나의 마진·클러치 판단 가능케.
- **(C) 시드 RNG 주입** — `index.html`의 `mulberry32`를 시뮬로 이식하고 시드 인자 주입. 페르소나 공정 A/B의 전제.

> 이 변경들은 기존 그리디 동작을 보존하는 방향(기본 pick = 기존 그리디)으로 한다.

### 복붙 부채 = 가드만 (단순성 우선)
규칙 5중 복붙(`connect`/`gain`/`evalHand`/`BOSSES`/`blindTarget`)은 근본 해결(engine 추출)이 `index.html`의 "외부라이브러리0·번들러0·빌드0" 핵심가치와 충돌하므로 **이번에는 미룬다**. 대신:
- Fun QA 엔진은 **run-sim.cjs에서만** 규칙을 가져온다(6번째 사본 최소화 — 가능하면 run-sim을 require).
- **드리프트 가드 테스트**: "Fun QA 엔진 규칙이 현재 run-sim 규칙과 일치하는가"를 `assert`로 고정해 밸런스 변경 시 자동 검출.

## 8. 구현 계획 — 3 Phase (greybox)

### Phase 1 — 정량 코어 (Layer 1 + 2)
- 선결 인프라 A/B/C.
- 페르소나 봇 5종(`tools/funqa/personas.cjs`).
- 재미 메트릭 5축 엔진(`tools/funqa/metrics.cjs`).
- 콘솔 리포트(`tools/funqa/run-funqa.cjs`).
- **검증**: 골든 테스트(§6) — 안테1 인색한손에서 주체성 메트릭 하락 재현. 드리프트 가드 통과.

### Phase 2 — LLM 판정관 (Layer 3)
- 페르소나별 LLM 판정관 + 분포 판정(70% 기준) + 편향 경고.
- **검증**: 알려진 좋은 빌드/나쁜 빌드에 대해 LLM 판정이 메트릭·사람 직관과 일치하는가(소수 골든 케이스).

### Phase 3 — 회귀 게이트 (운영)
- LLM 플레이테스터(think-aloud) 의심지점 검증.
- 버전 비교(직전 vs 현재) 재미 델타 + pass/fail 게이트.
- **검증**: 의도적으로 재미를 깎는 패치(예: 손패 2 고정)를 넣으면 게이트가 FAIL + 정확한 진단을 내는가.

## 9. 리스크 / 미해결 가설 (정직하게)

| 항목 | 상태 | 완화 |
|---|---|---|
| **봇이 사람 재미를 대표하는가** (근본 가설) | 미검증 | §6 골든 테스트 + 사람 텔레메트리 대조 캘리브레이션. 최종 판정자 아님(보조). |
| **메트릭 게이밍** (메트릭 최적화 ≠ 재미 최적화) | 위험 | 5축 균형 + LLM 정성 교차검증 + 골든 케이스 회귀. |
| **LLM 판정 신뢰성·편향** | 위험 | 페르소나 다양성, 메트릭과 교차, 비용 관리(의심지점만). |
| **5축이 실제 재미와 상관있나** | 미검증 가설 | 사람 로그 대조. 미달 시 메트릭 재설계(밸런스 튜닝 아님). |
| **70% 임계 적정성** | 미검증 | 초기값, 사람 데이터로 보정. |
| **LLM 비용·속도** | 관리 대상 | Layer 1+2 매번 무료, Layer 3는 주요 패치·의심지점만. |
